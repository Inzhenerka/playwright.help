## Настройка GitHub Actions

Чтобы добавить файл [GitHub Actions](https://docs.github.com/en/actions), сначала создайте папку `.github/workflows`, а затем добавьте в неё файл `playwright.yml` с примером кода ниже, чтобы ваши тесты запускались при каждом push и pull request в ветку main/master.

```yml title=".github/workflows/playwright.yml"
name: Playwright Tests
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v5
    - uses: actions/setup-java@v5
      with:
        distribution: 'temurin'
        java-version: '25'
    - name: Build & Install
      run: mvn -B install -D skipTests --no-transfer-progress
    - name: Ensure browsers are installed
      run: mvn exec:java -e -D exec.mainClass=com.microsoft.playwright.CLI -D exec.args="install --with-deps"
    - name: Run tests
      run: mvn test
```

Чтобы узнать больше об этом, см. ["Понимание GitHub Actions"](https://docs.github.com/en/actions/learn-github-actions/understanding-github-actions).

Если посмотреть на список шагов в `jobs.test.steps`, видно, что workflow выполняет следующие действия:
1. Клонирует ваш репозиторий
1. Устанавливает зависимости языка
1. Устанавливает зависимости проекта и выполняет сборку (build)
1. Устанавливает браузеры Playwright
1. Запускает тесты

## Создание репозитория и отправка на GitHub

После того как вы настроили [workflow GitHub Actions](#setting-up-github-actions), вам остаётся только [создать репозиторий на GitHub](https://docs.github.com/en/get-started/quickstart/create-a-repo) или запушить код в уже существующий репозиторий. Следуйте инструкциям на GitHub и не забудьте [инициализировать git-репозиторий](https://github.com/git-guides/git-init) командой `git init`, чтобы вы могли [добавлять](https://github.com/git-guides/git-add), [коммитить](https://github.com/git-guides/git-commit) и [пушить](https://github.com/git-guides/git-push) свой код.

[trace.playwright.dev](https://trace.playwright.dev) — это статически размещенный вариант Trace Viewer. Вы можете загружать файлы трассировки, используя перетаскивание.

![playwright trace viewer](../images/getting-started/trace-viewer-failed-test.png)

## Properly handling Secrets

Artifacts like trace files or console logs contain information about your test execution. They can contain sensitive data like user credentials for a test user, access tokens to a staging backend, testing source code, or sometimes even your application source code. Treat these files just as carefully as you treat that sensitive data. If you upload reports and traces as part of your CI workflow, make sure that you only upload them to trusted artifact stores, or that you encrypt the files before upload. The same is true for sharing artifacts with team members: Use a trusted file share or encrypt the files before sharing.
