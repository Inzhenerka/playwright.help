## Введение

Playwright Test выполняет тесты параллельно. Для этого он запускает несколько рабочих процессов, которые работают одновременно. По умолчанию **тестовые файлы** выполняются параллельно. Тесты в одном файле выполняются последовательно, в одном и том же рабочем процессе.
- Вы можете настроить тесты с помощью [`test.describe.configure`](#parallelize-tests-in-a-single-file), чтобы запускать **тесты в одном файле** параллельно.
- Вы можете настроить **весь проект** так, чтобы все тесты во всех файлах выполнялись параллельно, используя [testProject.fullyParallel](/api/class-testproject.mdx#test-project-fully-parallel) или [testConfig.fullyParallel](/api/class-testconfig.mdx#test-config-fully-parallel).
- Чтобы **отключить** параллелизм, ограничьте количество [рабочих процессов до одного](#disable-parallelism).

Вы можете контролировать количество [параллельных рабочих процессов](#limit-workers) и [ограничить количество сбоев](#limit-failures-and-fail-fast) в целом тестовом наборе для повышения эффективности.

## Рабочие процессы

Все тесты выполняются в рабочих процессах. Эти процессы являются процессами ОС, работающими независимо, управляемыми тестовым раннером. Все рабочие процессы имеют идентичные окружения и каждый запускает свой собственный браузер.

Вы не можете общаться между рабочими процессами. Playwright Test повторно использует один рабочий процесс столько, сколько может, чтобы ускорить тестирование, поэтому несколько тестовых файлов обычно выполняются в одном рабочем процессе один за другим.

Рабочие процессы всегда завершаются после [сбоя теста](./test-retries.mdx#failures), чтобы гарантировать чистую среду для последующих тестов.

## Ограничение рабочих процессов

Вы можете контролировать максимальное количество параллельных рабочих процессов через [командную строку](./test-cli.mdx) или в [файле конфигурации](./test-configuration.mdx).

Из командной строки:

```bash
npx playwright test --workers 4
```

В файле конфигурации:

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  // Ограничьте количество рабочих процессов на CI, используйте значение по умолчанию локально
  workers: process.env.CI ? 2 : undefined,
});
```

## Отключение параллелизма

Вы можете отключить любой параллелизм, разрешив только один рабочий процесс в любое время. Либо установите опцию `workers: 1` в файле конфигурации, либо передайте `--workers=1` в командной строке.

```bash
npx playwright test --workers=1
```

## Параллелизация тестов в одном файле

По умолчанию тесты в одном файле выполняются последовательно. Если у вас есть много независимых тестов в одном файле, вы можете захотеть запустить их параллельно с помощью [test.describe.configure()](/api/class-test.mdx#test-describe-configure).

Обратите внимание, что параллельные тесты выполняются в отдельных рабочих процессах и не могут делиться никаким состоянием или глобальными переменными. Каждый тест выполняет все соответствующие хуки только для себя, включая `beforeAll` и `afterAll`.

```js
import { test } from '@playwright/test';

test.describe.configure({ mode: 'parallel' });

test('runs in parallel 1', async ({ page }) => { /* ... */ });
test('runs in parallel 2', async ({ page }) => { /* ... */ });
```

В качестве альтернативы вы можете включить все тесты в этот полностью параллельный режим в файле конфигурации:

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  fullyParallel: true,
});
```

Вы также можете включить полностью параллельный режим только для нескольких проектов:

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  // запускает все тесты во всех файлах конкретного проекта параллельно
  projects: [
    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'] },
      fullyParallel: true,
    },
  ]
});
```

## Последовательный режим

Вы можете аннотировать взаимозависимые тесты как последовательные. Если один из последовательных тестов завершится неудачей, все последующие тесты будут пропущены. Все тесты в группе повторяются вместе.

:::note

Использование последовательного режима не рекомендуется. Обычно лучше сделать ваши тесты изолированными, чтобы их можно было запускать независимо.
:::

```js
import { test, type Page } from '@playwright/test';

// Аннотируйте весь файл как последовательный.
test.describe.configure({ mode: 'serial' });

let page: Page;

test.beforeAll(async ({ browser }) => {
  page = await browser.newPage();
});

test.afterAll(async () => {
  await page.close();
});

test('runs first', async () => {
  await page.goto('https://playwright.dev/');
});

test('runs second', async () => {
  await page.getByText('Get Started').click();
});
```

## Разделение тестов между несколькими машинами

Playwright Test может разделить тестовый набор, чтобы его можно было выполнять на нескольких машинах. См. [руководство по шардированию](./test-sharding.mdx) для получения дополнительных сведений.

```bash
npx playwright test --shard=2/3
```

## Ограничение сбоев и быстрая остановка

Вы можете ограничить количество неудачных тестов в целом тестовом наборе, установив опцию конфигурации `maxFailures` или передав флаг командной строки `--max-failures`.

При запуске с установленным "максимальным количеством сбоев" Playwright Test остановится после достижения этого числа неудачных тестов и пропустит любые тесты, которые еще не были выполнены. Это полезно, чтобы избежать растраты ресурсов на сломанные тестовые наборы.

Передача параметра командной строки:

```bash
npx playwright test --max-failures=10
```

Установка в файле конфигурации:

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  // Ограничьте количество сбоев на CI, чтобы сэкономить ресурсы
  maxFailures: process.env.CI ? 10 : undefined,
});
```

## Индекс рабочего процесса и параллельный индекс

Каждому рабочему процессу присваиваются два идентификатора: уникальный индекс рабочего процесса, который начинается с 1, и параллельный индекс, который находится между `0` и `workers - 1`. Когда рабочий процесс перезапускается, например, после сбоя, новый рабочий процесс имеет тот же `parallelIndex` и новый `workerIndex`.

Вы можете прочитать индекс из переменных окружения `process.env.TEST_WORKER_INDEX` и `process.env.TEST_PARALLEL_INDEX`, или получить к ним доступ через [testInfo.workerIndex](/api/class-testinfo.mdx#test-info-worker-index) и [testInfo.parallelIndex](/api/class-testinfo.mdx#test-info-parallel-index).

### Изоляция тестовых данных между параллельными рабочими процессами

Вы можете использовать `process.env.TEST_WORKER_INDEX` или [testInfo.workerIndex](/api/class-testinfo.mdx#test-info-worker-index), упомянутый выше, чтобы изолировать пользовательские данные в базе данных между тестами, выполняемыми на разных рабочих процессах. Все тесты, выполняемые рабочим процессом, повторно используют одного и того же пользователя.

Создайте файл `playwright/fixtures.ts`, который будет [создавать фикстуру `dbUserName`](./test-fixtures#creating-a-fixture) и инициализировать нового пользователя в тестовой базе данных. Используйте [testInfo.workerIndex](/api/class-testinfo.mdx#test-info-worker-index), чтобы различать рабочих процессов.

```js title="playwright/fixtures.ts"
import { test as baseTest, expect } from '@playwright/test';
// Импортируйте утилиты проекта для управления пользователями в тестовой базе данных.
import { createUserInTestDatabase, deleteUserFromTestDatabase } from './my-db-utils';

export * from '@playwright/test';
export const test = baseTest.extend<{}, { dbUserName: string }>({
  // Возвращает имя пользователя базы данных, уникальное для рабочего процесса.
  dbUserName: [async ({ }, use) => {
    // Используйте workerIndex как уникальный идентификатор для каждого рабочего процесса.
    const userName = `user-${test.info().workerIndex}`;
    // Инициализируйте пользователя в базе данных.
    await createUserInTestDatabase(userName);
    await use(userName);
    // Очистите после завершения тестов.
    await deleteUserFromTestDatabase(userName);
  }, { scope: 'worker' }],
});
```

Теперь каждый тестовый файл должен импортировать `test` из нашего файла фикстур вместо `@playwright/test`.

```js title="tests/example.spec.ts"
// Важно: импортируйте наши фикстуры.
import { test, expect } from '../playwright/fixtures';

test('test', async ({ dbUserName }) => {
  // Используйте имя пользователя в тесте.
});
```

## Управление порядком тестов

Playwright Test выполняет тесты из одного файла в порядке их объявления, если вы не [параллелизуете тесты в одном файле](#parallelize-tests-in-a-single-file).

Нет гарантии относительно порядка выполнения тестов между файлами, поскольку Playwright Test по умолчанию выполняет тестовые файлы параллельно. Однако, если вы [отключите параллелизм](#disable-parallelism), вы можете контролировать порядок тестов, либо назвав ваши файлы в алфавитном порядке, либо используя файл "списка тестов".

### Сортировка тестовых файлов в алфавитном порядке

Когда вы **отключаете параллельное выполнение тестов**, Playwright Test выполняет тестовые файлы в алфавитном порядке. Вы можете использовать некоторую конвенцию именования, чтобы контролировать порядок тестов, например `001-user-signin-flow.spec.ts`, `002-create-new-document.spec.ts` и так далее.

### Использование файла "списка тестов"

:::warning

Списки тестов не рекомендуются и поддерживаются только на основе лучших усилий. Некоторые функции, такие как расширение VS Code и трассировка, могут работать неправильно со списками тестов.
:::

Вы можете поместить ваши тесты в вспомогательные функции в нескольких файлах. Рассмотрим следующий пример, где тесты не определены непосредственно в файле, а определены в обертке функции.

```js title="feature-a.spec.ts"
import { test, expect } from '@playwright/test';

export default function createTests() {
  test('feature-a example test', async ({ page }) => {
    // ... тест здесь
  });
}

```

```js title="feature-b.spec.ts"
import { test, expect } from '@playwright/test';

export default function createTests() {
  test.use({ viewport: { width: 500, height: 500 } });

  test('feature-b example test', async ({ page }) => {
    // ... тест здесь
  });
}
```

Вы можете создать файл списка тестов, который будет контролировать порядок тестов - сначала запустите тесты `feature-b`, затем тесты `feature-a`. Обратите внимание, что каждый тестовый файл обернут в блок `test.describe()`, который вызывает функцию, где определены тесты. Таким образом, вызовы `test.use()` влияют только на тесты из одного файла.

```js title="test.list.ts"
import { test } from '@playwright/test';
import featureBTests from './feature-b.spec.ts';
import featureATests from './feature-a.spec.ts';

test.describe(featureBTests);
test.describe(featureATests);
```

Теперь **отключите параллельное выполнение**, установив количество рабочих процессов равным одному, и укажите ваш файл списка тестов.

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  workers: 1,
  testMatch: 'test.list.ts',
});
```

:::note
Не определяйте ваши тесты непосредственно в вспомогательном файле. Это может привести к неожиданным результатам, поскольку ваши тесты теперь зависят от порядка операторов `import`/`require`. Вместо этого оберните тесты в функцию, которая будет явно вызвана файлом списка тестов, как в приведенном выше примере.
:::
