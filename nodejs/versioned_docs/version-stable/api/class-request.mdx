Каждый раз, когда страница отправляет запрос на сетевой ресурс, происходит следующая последовательность событий, генерируемых [Page]:
* [page.on('request')](/api/class-page.mdx#page-event-request) генерируется, когда запрос инициируется страницей.
* [page.on('response')](/api/class-page.mdx#page-event-response) генерируется, когда/если статус ответа и заголовки получены для запроса.
* [page.on('requestfinished')](/api/class-page.mdx#page-event-request-finished) генерируется, когда тело ответа загружено и запрос завершен.

Если запрос не удался на каком-либо этапе, то вместо события `'requestfinished'` (и, возможно, вместо события 'response') генерируется событие [page.on('requestfailed')](/api/class-page.mdx#page-event-request-failed).

:::note

HTTP-ошибки, такие как 404 или 503, все еще считаются успешными ответами с точки зрения HTTP, поэтому запрос завершится с событием `'requestfinished'`.
:::

Если запрос получает ответ 'redirect', запрос успешно завершается с событием `requestfinished`, и новый запрос отправляется на перенаправленный URL.

### failure {#request-failure}

<font size="2" style={{position: "relative", top: "-20px"}}>Добавлено до v1.9</font><x-search>request.failure</x-search>

Метод возвращает `null`, если этот запрос не завершился неудачей, как сообщается событием `requestfailed`.

**Использование**

Пример логирования всех неудачных запросов:

```js
page.on('requestfailed', request => {
  console.log(request.url() + ' ' + request.failure().errorText);
});
```

**Возвращает**
- [null] | [Object]<a aria-hidden="true" tabIndex="-1" class="list-anchor-link" id="request-failure-return"/><a href="#request-failure-return" class="list-anchor">#</a>
  - `errorText` [string]
    
    Читаемое сообщение об ошибке, например, `'net::ERR_FAILED'`.

### frame {#request-frame}

<font size="2" style={{position: "relative", top: "-20px"}}>Добавлено до v1.9</font><x-search>request.frame</x-search>

Возвращает [Frame], который инициировал этот запрос.

**Использование**

```js
const frameUrl = request.frame().url();
```

**Возвращает**
- [Frame]<a aria-hidden="true" tabIndex="-1" class="list-anchor-link" id="request-frame-return"/><a href="#request-frame-return" class="list-anchor">#</a>

**Подробности**

Обратите внимание, что в некоторых случаях фрейм может быть недоступен, и этот метод вызовет ошибку.
* Когда запрос исходит от Service Worker. Вы можете использовать `request.serviceWorker()`, чтобы это проверить.
* Когда запрос на навигацию инициируется до создания соответствующего фрейма. Вы можете использовать [request.isNavigationRequest()](/api/class-request.mdx#request-is-navigation-request), чтобы это проверить.

Вот пример, который обрабатывает все случаи:

```js
if (request.serviceWorker())
  console.log(`request ${request.url()} from a service worker`);
else if (request.isNavigationRequest())
  console.log(`request ${request.url()} is a navigation request`);
else
  console.log(`request ${request.url()} from a frame ${request.frame().url()}`);
```

### redirectedFrom {#request-redirected-from}

<font size="2" style={{position: "relative", top: "-20px"}}>Добавлено до v1.9</font><x-search>request.redirectedFrom</x-search>

Запрос, который был перенаправлен сервером на этот, если таковой имеется.

Когда сервер отвечает с перенаправлением, Playwright создает новый объект [Request]. Эти два запроса связаны методами `redirectedFrom()` и `redirectedTo()`. Когда произошло несколько перенаправлений сервера, можно построить всю цепочку перенаправлений, повторно вызывая `redirectedFrom()`.

**Использование**

Например, если веб-сайт `http://example.com` перенаправляет на `https://example.com`:

```js
const response = await page.goto('http://example.com');
console.log(response.request().redirectedFrom().url()); // 'http://example.com'
```

Если веб-сайт `https://google.com` не имеет перенаправлений:

```js
const response = await page.goto('https://google.com');
console.log(response.request().redirectedFrom()); // null
```

**Возвращает**
- [null] | [Request]<a aria-hidden="true" tabIndex="-1" class="list-anchor-link" id="request-redirected-from-return"/><a href="#request-redirected-from-return" class="list-anchor">#</a>

### timing {#request-timing}

<font size="2" style={{position: "relative", top: "-20px"}}>Добавлено до v1.9</font><x-search>request.timing</x-search>

Возвращает информацию о времени ресурса для данного запроса. Большинство значений времени становятся доступными после получения ответа, `responseEnd` становится доступным, когда запрос завершается. Найдите дополнительную информацию в [API временных ресурсов](https://developer.mozilla.org/en-US/docs/Web/API/PerformanceResourceTiming).

**Использование**

```js
const requestFinishedPromise = page.waitForEvent('requestfinished');
await page.goto('http://example.com');
const request = await requestFinishedPromise;
console.log(request.timing());
```

**Возвращает**
- [Object]<a aria-hidden="true" tabIndex="-1" class="list-anchor-link" id="request-timing-return"/><a href="#request-timing-return" class="list-anchor">#</a>
  - `startTime` [number]
    
    Время начала запроса в миллисекундах, прошедших с 1 января 1970 года 00:00:00 UTC
  - `domainLookupStart` [number]
    
    Время сразу перед тем, как браузер начинает поиск доменного имени для ресурса. Значение указывается в миллисекундах относительно `startTime`, -1, если недоступно.
  - `domainLookupEnd` [number]
    
    Время сразу после того, как браузер начинает поиск доменного имени для ресурса. Значение указывается в миллисекундах относительно `startTime`, -1, если недоступно.
  - `connectStart` [number]
    
    Время сразу перед тем, как пользовательский агент начинает устанавливать соединение с сервером для получения ресурса. Значение указывается в миллисекундах относительно `startTime`, -1, если недоступно.
  - `secureConnectionStart` [number]
    
    Время сразу перед тем, как браузер начинает процесс рукопожатия для обеспечения текущего соединения. Значение указывается в миллисекундах относительно `startTime`, -1, если недоступно.
  - `connectEnd` [number]
    
    Время сразу перед тем, как пользовательский агент начинает устанавливать соединение с сервером для получения ресурса. Значение указывается в миллисекундах относительно `startTime`, -1, если недоступно.
  - `requestStart` [number]
    
    Время сразу перед тем, как браузер начинает запрашивать ресурс у сервера, кэша или локального ресурса. Значение указывается в миллисекундах относительно `startTime`, -1, если недоступно.
  - `responseStart` [number]
    
    Время сразу после того, как браузер получает первый байт ответа от сервера, кэша или локального ресурса. Значение указывается в миллисекундах относительно `startTime`, -1, если недоступно.
  - `responseEnd` [number]
    
    Время сразу после того, как браузер получает последний байт ресурса или сразу перед закрытием транспортного соединения, в зависимости от того, что произойдет первым. Значение указывается в миллисекундах относительно `startTime`, -1, если недоступно.
