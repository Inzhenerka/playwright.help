## Обзор

С помощью тестирования снимков в Playwright вы можете проверить дерево доступности страницы на соответствие заранее определенному шаблону снимка.

```js
await page.goto('https://playwright.dev/');
await expect(page.getByRole('banner')).toMatchAriaSnapshot(`
  - banner:
    - heading /Playwright enables reliable end-to-end/ [level=1]
    - link "Get started"
    - link "Star microsoft/playwright on GitHub"
    - link /[\\d]+k\\+ stargazers on GitHub/
`);
```

## Тестирование утверждений против тестирования снимков

Тестирование снимков и тестирование утверждений служат разным целям в автоматизации тестирования:

### Тестирование утверждений

Тестирование утверждений — это целенаправленный подход, при котором вы проверяете конкретные значения или условия для элементов или компонентов. Например, в Playwright, [expect(locator).toHaveText()](/api/class-locatorassertions.mdx#locator-assertions-to-have-text) проверяет, что элемент содержит ожидаемый текст, а [expect(locator).toHaveValue()](/api/class-locatorassertions.mdx#locator-assertions-to-have-value) подтверждает, что поле ввода имеет ожидаемое значение. Тесты утверждений специфичны и, как правило, проверяют текущее состояние элемента или свойства на соответствие ожидаемому, заранее определенному состоянию. Они хорошо работают для предсказуемых проверок с одним значением, но имеют ограничения при тестировании более широкой структуры или вариаций.

**Преимущества**
- **Ясность**: Намерение теста явно и легко понимается.
- **Специфичность**: Тесты сосредоточены на конкретных аспектах функциональности, что делает их более устойчивыми к несвязанным изменениям.
- **Отладка**: Ошибки предоставляют целенаправленную обратную связь, указывая непосредственно на проблемный аспект.

**Недостатки**
- **Многословность для сложных выводов**: Написание утверждений для сложных структур данных или больших выводов может быть громоздким и подверженным ошибкам.
- **Затраты на обслуживание**: По мере изменения кода ручное обновление утверждений может занять много времени.

### Тестирование снимков

Тестирование снимков захватывает "снимок" или представление всего состояния элемента, компонента или данных в данный момент, который затем сохраняется для будущих сравнений. При повторном запуске тестов текущее состояние сравнивается с снимком, и если есть различия, тест не проходит. Этот подход особенно полезен для сложных или динамических структур, где ручная проверка каждой детали заняла бы слишком много времени. Тестирование снимков более широкое и целостное, чем тестирование утверждений, позволяя отслеживать более сложные изменения с течением времени.

**Преимущества**
- **Упрощает сложные выводы**: Например, тестирование отрисованного вывода UI-компонента может быть утомительным с традиционными утверждениями. Снимки захватывают весь вывод для легкого сравнения.
- **Быстрая обратная связь**: Разработчики могут легко заметить непреднамеренные изменения в выводе.
- **Стимулирует согласованность**: Помогает поддерживать согласованный вывод по мере изменения кода.

**Недостатки**
- **Чрезмерная зависимость**: Может возникнуть соблазн принимать изменения в снимках, не понимая их полностью, что может скрывать ошибки.
- **Гранулярность**: Большие снимки могут быть трудны для интерпретации, когда возникают различия, особенно если незначительные изменения затрагивают большие части вывода.
- **Подходящесть**: Не идеальны для высокодинамичного контента, где выводы меняются часто или непредсказуемо.

### Когда использовать
- **Тестирование снимков** идеально подходит для:
  - Тестирования UI целых страниц и компонентов.
  - Широких структурных проверок для сложных UI-компонентов.
  - Регрессионного тестирования для выводов, которые редко меняют структуру.
- **Тестирование утверждений** идеально подходит для:
  - Валидации основной логики.
  - Тестирования вычисленных значений.
  - Тестов с высокой точностью, требующих точных условий.

Сочетая тестирование снимков для широких структурных проверок и тестирование утверждений для конкретной функциональности, вы можете достичь сбалансированной стратегии тестирования.

## Aria снимки

В Playwright aria снимки предоставляют YAML-представление дерева доступности страницы. Эти снимки могут быть сохранены и сравнены позже, чтобы проверить, остается ли структура страницы последовательной или соответствует установленным ожиданиям.

Формат YAML описывает иерархическую структуру доступных элементов на странице, детализируя **роли**, **атрибуты**, **значения** и **текстовое содержимое**. Структура следует синтаксису, похожему на дерево, где каждый узел представляет доступный элемент, а отступы указывают на вложенные элементы.

Каждый доступный элемент в дереве представлен как узел YAML:

```yaml
- role "name" [attribute=value]
```

- **role**: Указывает ARIA или HTML роль элемента (например, `heading`, `list`, `listitem`, `button`).
- **"name"**: Доступное имя элемента. Цитируемые строки указывают на точные значения, а `/patterns/` используются для регулярных выражений.
- **[attribute=value]**: Атрибуты и значения, в квадратных скобках, представляют собой конкретные ARIA атрибуты, такие как `checked`, `disabled`, `expanded`, `level`, `pressed` или `selected`.

Эти значения получены из ARIA атрибутов или вычислены на основе HTML семантики. Чтобы исследовать структуру дерева доступности страницы, используйте [панель доступности Chrome DevTools](https://developer.chrome.com/docs/devtools/accessibility/reference#pane).

## Сравнение снимков

Метод утверждения [expect(locator).toMatchAriaSnapshot()](/api/class-locatorassertions.mdx#locator-assertions-to-match-aria-snapshot) в Playwright сравнивает доступную структуру области локатора с заранее определенным шаблоном aria снимка, помогая проверить состояние страницы на соответствие требованиям тестирования.

Для следующего DOM:

```html
<h1>title</h1>
```

Вы можете сопоставить его с помощью следующего шаблона снимка:

```js
await expect(page.locator('body')).toMatchAriaSnapshot(`
  - heading "title"
`);
```

При сравнении шаблон снимка сопоставляется с текущим деревом доступности страницы:
* Если структура дерева соответствует шаблону, тест проходит; в противном случае он не проходит, указывая на несоответствие между ожидаемыми и фактическими состояниями доступности.
* Сравнение чувствительно к регистру и игнорирует пробелы, поэтому отступы и переносы строк игнорируются.
* Сравнение чувствительно к порядку, что означает, что порядок элементов в шаблоне снимка должен соответствовать порядку в дереве доступности страницы.

### Частичное соответствие

Вы можете выполнять частичные совпадения узлов, опуская атрибуты или доступные имена, что позволяет проверять конкретные части дерева доступности без необходимости точного соответствия. Эта гибкость полезна для динамических или несущественных атрибутов.

```html
<button>Submit</button>
```

*aria снимок*

```yaml
- button
```

В этом примере роль кнопки сопоставляется, но доступное имя ("Submit") не указано, что позволяет тесту пройти независимо от метки кнопки.

<hr/>

Для элементов с ARIA атрибутами, такими как `checked` или `disabled`, опускание этих атрибутов позволяет частичное соответствие, сосредотачиваясь исключительно на роли и иерархии.

```html
<input type="checkbox" checked>
```

*aria снимок для частичного соответствия*

```yaml
- checkbox
```

В этом частичном соответствии атрибут `checked` игнорируется, поэтому тест пройдет независимо от состояния флажка.

<hr/>

Аналогично, вы можете частично сопоставлять дочерние элементы в списках или группах, опуская конкретные элементы списка или вложенные элементы.

```html
<ul>
  <li>Feature A</li>
  <li>Feature B</li>
  <li>Feature C</li>
</ul>
```

*aria снимок для частичного соответствия*

```yaml
- list
  - listitem: Feature B
```

Частичные совпадения позволяют создавать гибкие тесты снимков, которые проверяют важную структуру страницы без необходимости навязывать конкретное содержимое или атрибуты.

### Сопоставление с регулярными выражениями

Регулярные выражения позволяют гибко сопоставлять элементы с динамическим или переменным текстом. Доступные имена и текст могут поддерживать шаблоны regex.

```html
<h1>Issues 12</h1>
```

*aria снимок с регулярным выражением*

```yaml
- heading /Issues \d+/
```

## Генерация снимков

Создание aria снимков в Playwright помогает обеспечить и поддерживать структуру вашего приложения. Вы можете генерировать снимки различными способами в зависимости от вашей настройки тестирования и рабочего процесса.

### Генерация снимков с помощью генератора кода Playwright

Если вы используете [Генератор кода Playwright](./codegen.mdx), создание aria снимков упрощается с его интерактивным интерфейсом:
- **Действие "Утверждать снимок"**: В генераторе кода вы можете использовать действие "Утверждать снимок", чтобы автоматически создать утверждение снимка для выбранных элементов. Это быстрый способ захватить aria снимок как часть вашего записанного тестового потока.
- **Вкладка "Aria снимок"**: Вкладка "Aria снимок" в интерфейсе генератора кода визуально представляет aria снимок для выбранного локатора, позволяя вам исследовать, проверять и подтверждать роли элементов, атрибуты и доступные имена для помощи в создании и проверке снимков.

### Обновление снимков с помощью `@playwright/test` и флага `--update-snapshots`

При использовании тестового раннера Playwright (`@playwright/test`) вы можете автоматически обновлять снимки с помощью флага `--update-snapshots`, `-u` для краткости.

Запуск тестов с флагом `--update-snapshots` обновит снимки, которые не совпадают. Совпадающие снимки обновляться не будут.

```bash
npx playwright test --update-snapshots
```

Обновление снимков полезно, когда изменения в структуре приложения требуют новых снимков в качестве базового уровня. Обратите внимание, что Playwright будет ждать максимального времени ожидания expect, указанного в конфигурации тестового раннера, чтобы убедиться, что страница стабилизировалась перед созданием снимка. Возможно, потребуется настроить `--timeout`, если тест достигает времени ожидания при создании снимков.

#### Пустой шаблон для генерации снимков

Передача пустой строки в качестве шаблона в утверждении генерирует снимок на лету:

```js
await expect(locator).toMatchAriaSnapshot('');
```

Обратите внимание, что Playwright будет ждать максимального времени ожидания expect, указанного в конфигурации тестового раннера, чтобы убедиться, что страница стабилизировалась перед созданием снимка. Возможно, потребуется настроить `--timeout`, если тест достигает времени ожидания при создании снимков.

#### Файлы патчей снимков

При обновлении снимков Playwright создает файлы патчей, которые фиксируют различия. Эти файлы патчей могут быть просмотрены, применены и зафиксированы в системе контроля версий, позволяя командам отслеживать структурные изменения с течением времени и обеспечивать согласованность обновлений с требованиями приложения.

Способ обновления исходного кода можно изменить с помощью флага `--update-source-method`. Доступно несколько вариантов:
- **"patch"** (по умолчанию): Генерирует файл унифицированного диффа, который можно применить к исходному коду с помощью `git apply`.
- **"3way"**: Генерирует маркеры конфликтов слияния в вашем исходном коде, позволяя вам выбрать, принимать ли изменения.
- **"overwrite"**: Перезаписывает исходный код новыми значениями снимка.

```bash
npx playwright test --update-snapshots --update-source-mode=3way
```

#### Снимки как отдельные файлы

Чтобы сохранить ваши снимки в отдельном файле, используйте метод `toMatchAriaSnapshot` с опцией `name`, указывая расширение `.aria.yml`.

```js
await expect(page.getByRole('main')).toMatchAriaSnapshot({ name: 'main.aria.yml' });
```

По умолчанию снимки из тестового файла `example.spec.ts` помещаются в директорию `example.spec.ts-snapshots`. Поскольку снимки должны быть одинаковыми для всех браузеров, сохраняется только один снимок, даже если тестируется с несколькими браузерами. Если вы хотите, вы можете настроить [шаблон пути снимка](./api/class-testconfig#test-config-snapshot-path-template) с помощью следующей конфигурации:

```js
export default defineConfig({
  expect: {
    toMatchAriaSnapshot: {
      pathTemplate: '__snapshots__/{testFilePath}/{arg}{ext}',
    },
  },
});
```

### Использование метода `Locator.ariaSnapshot`

Метод [locator.ariaSnapshot()](/api/class-locator.mdx#locator-aria-snapshot) позволяет вам программно создать YAML-представление доступных элементов в области локатора, что особенно полезно для динамической генерации снимков во время выполнения теста.

**Пример**:

```js
const snapshot = await page.locator('body').ariaSnapshot();
console.log(snapshot);
```

Эта команда выводит aria снимок в области указанного локатора в формате YAML, который вы можете проверить или сохранить по мере необходимости.

## Примеры дерева доступности

### Заголовки с атрибутами уровня

Заголовки могут включать атрибут `level`, указывающий их уровень заголовка.

```html
<h1>Title</h1>
<h2>Subtitle</h2>
```

*aria снимок*

```yaml
- heading "Title" [level=1]
- heading "Subtitle" [level=2]
```

### Текстовые узлы

Отдельные или описательные текстовые элементы появляются как текстовые узлы.

```html
<div>Sample accessible name</div>
```

*aria снимок*

```yaml
- text: Sample accessible name
```

### Встроенный многострочный текст

Многострочный текст, такой как абзацы, нормализуется в aria снимке.

```html
<p>Line 1<br>Line 2</p>
```

*aria снимок*

```yaml
- paragraph: Line 1 Line 2
```

### Ссылки

Ссылки отображают свой текст или составное содержимое из псевдоэлементов.

```html
<a href="#more-info">Read more about Accessibility</a>
```

*aria снимок*

```yaml
- link "Read more about Accessibility"
```

### Текстовые поля

Элементы ввода типа `text` показывают содержимое своего атрибута `value`.

```html
<input type="text" value="Enter your name">
```

*aria снимок*

```yaml
- textbox: Enter your name
```

### Списки с элементами

Упорядоченные и неупорядоченные списки включают свои элементы списка.

```html
<ul aria-label="Main Features">
  <li>Feature 1</li>
  <li>Feature 2</li>
</ul>
```

*aria снимок*

```yaml
- list "Main Features":
  - listitem: Feature 1
  - listitem: Feature 2
```

### Группированные элементы

Группы захватывают вложенные элементы, такие как элементы `<details>` с содержимым резюме.

```html
<details>
  <summary>Summary</summary>
  <p>Detail content here</p>
</details>
```

*aria снимок*

```yaml
- group: Summary
```

### Атрибуты и состояния

Распространенные ARIA атрибуты, такие как `checked`, `disabled`, `expanded`, `level`, `pressed` и `selected`, представляют состояния управления.

#### Флажок с атрибутом `checked`

```html
<input type="checkbox" checked>
```

*aria снимок*

```yaml
- checkbox [checked]
```

#### Кнопка с атрибутом `pressed`

```html
<button aria-pressed="true">Toggle</button>
```

*aria снимок*

```yaml
- button "Toggle" [pressed=true]
```
