## Введение

Тесты Playwright могут выполняться в средах CI. Мы создали пример конфигураций для распространенных CI-провайдеров.

3 шага, чтобы запустить ваши тесты в CI:
1. **Убедитесь, что агент CI может запускать браузеры**: Используйте [наш образ Docker](./docker.mdx) на Linux-агентах или установите ваши зависимости с помощью [CLI](./browsers#install-system-dependencies).
1. **Установите Playwright**:
   
   ```bash
   # Установите пакеты NPM
   npm ci
   
   # Установите браузеры и зависимости Playwright
   npx playwright install --with-deps
   ```
   
1. **Запустите ваши тесты**:
   
   ```bash
   npx playwright test
   ```
   
## Рабочие процессы

Мы рекомендуем устанавливать [workers](./api/class-testconfig.mdx#test-config-workers) на "1" в средах CI, чтобы приоритизировать стабильность и воспроизводимость. Последовательное выполнение тестов гарантирует, что каждый тест получает все системные ресурсы, избегая потенциальных конфликтов. Однако, если у вас мощная самохостинговая CI-система, вы можете включить [параллельные](./test-parallel.mdx) тесты. Для более широкой параллелизации рассмотрите [шардинг](./test-parallel.mdx#shard-tests-between-multiple-machines) - распределение тестов между несколькими заданиями CI.

```js title="playwright.config.ts"
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  // Отказ от параллельных тестов в CI.
  workers: process.env.CI ? 1 : undefined,
});
```

## Конфигурации CI

[Инструменты командной строки](./browsers#install-system-dependencies) могут быть использованы для установки всех зависимостей операционной системы в CI.

### GitHub Actions

#### При push/pull_request

Тесты будут выполняться при push или pull request на ветках main/master. [Рабочий процесс](https://docs.github.com/en/actions/using-workflows/about-workflows) установит все зависимости, установит Playwright, а затем запустит тесты. Он также создаст HTML-отчет.

```yml title=".github/workflows/playwright.yml"
name: Playwright Tests
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: lts/*
    - name: Установить зависимости
      run: npm ci
    - name: Установить браузеры Playwright
      run: npx playwright install --with-deps
    - name: Запустить тесты Playwright
      run: npx playwright test
    - uses: actions/upload-artifact@v4
      if: ${{ !cancelled() }}
      with:
        name: playwright-report
        path: playwright-report/
        retention-days: 30
```

#### При push/pull_request (шардинг)

GitHub Actions поддерживает [шардинг тестов между несколькими заданиями](https://docs.github.com/en/actions/using-jobs/using-a-matrix-for-your-jobs). Ознакомьтесь с нашей [документацией по шардированию](./test-sharding), чтобы узнать больше о шардировании и увидеть [пример GitHub Actions](./test-sharding.mdx#github-actions-example) о том, как настроить задание для запуска ваших тестов на нескольких машинах, а также как объединить HTML-отчеты.

#### Через контейнеры

GitHub Actions поддерживает [запуск заданий в контейнере](https://docs.github.com/en/actions/using-jobs/running-jobs-in-a-container) с помощью опции [`jobs.<job_id>.container`](https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#jobsjob_idcontainer). Это полезно, чтобы не загрязнять хост-окружение зависимостями и иметь согласованное окружение для, например, тестирования скриншотов/визуальной регрессии на разных операционных системах.

```yml title=".github/workflows/playwright.yml"
name: Playwright Tests
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
jobs:
  playwright:
    name: 'Playwright Tests'
    runs-on: ubuntu-latest
    container:
      image: mcr.microsoft.com/playwright:v1.51.0-noble
      options: --user 1001
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: lts/*
      - name: Установить зависимости
        run: npm ci
      - name: Запустить ваши тесты
        run: npx playwright test
```

#### При развертывании

Это запустит тесты после того, как [развертывание GitHub](https://developer.github.com/v3/repos/deployments/) перейдет в состояние `success`. Сервисы, такие как Vercel, используют этот шаблон, чтобы вы могли запускать свои end-to-end тесты в их развернутом окружении.

```yml title=".github/workflows/playwright.yml"
name: Playwright Tests
on:
  deployment_status:
jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    if: github.event.deployment_status.state == 'success'
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: lts/*
    - name: Установить зависимости
      run: npm ci
    - name: Установить Playwright
      run: npx playwright install --with-deps
    - name: Запустить тесты Playwright
      run: npx playwright test
      env:
        PLAYWRIGHT_TEST_BASE_URL: ${{ github.event.deployment_status.target_url }}
```

#### Fail-Fast

Большие наборы тестов могут занимать очень много времени для выполнения. Выполняя предварительный запуск тестов с флагом `--only-changed`, вы можете сначала запустить файлы тестов, которые, вероятно, провалятся. Это даст вам более быстрый цикл обратной связи и немного снизит потребление CI во время работы с Pull Requests. Чтобы обнаружить файлы тестов, затронутые вашим изменением, `--only-changed` анализирует граф зависимостей ваших наборов. Это эвристика и может пропустить тесты, поэтому важно всегда запускать полный набор тестов после предварительного запуска тестов.

```yml title=".github/workflows/playwright.yml" {24-26}
name: Playwright Tests
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
      with:
        # Принудительно выполнить неповерхностный checkout, чтобы мы могли ссылаться на $GITHUB_BASE_REF.
        # См. https://github.com/actions/checkout для получения дополнительной информации.
        fetch-depth: 0
    - uses: actions/setup-node@v4
      with:
        node-version: lts/*
    - name: Установить зависимости
      run: npm ci
    - name: Установить браузеры Playwright
      run: npx playwright install --with-deps
    - name: Запустить измененные тесты Playwright
      run: npx playwright test --only-changed=$GITHUB_BASE_REF
      if: github.event_name == 'pull_request'
    - name: Запустить тесты Playwright
      run: npx playwright test
    - uses: actions/upload-artifact@v4
      if: ${{ !cancelled() }}
      with:
        name: playwright-report
        path: playwright-report/
        retention-days: 30
```

### Docker

У нас есть [предварительно собранный образ Docker](./docker.mdx), который можно использовать напрямую или как ссылку для обновления ваших существующих определений Docker.

Рекомендуемая конфигурация
1. Использование `--ipc=host` также рекомендуется при использовании Chromium. Без этого Chromium может исчерпать память и аварийно завершиться. Узнайте больше об этой опции в [документации Docker](https://docs.docker.com/reference/cli/docker/container/run/#ipc).
1. Видите другие странные ошибки при запуске Chromium? Попробуйте запустить ваш контейнер с `docker run --cap-add=SYS_ADMIN`, когда разрабатываете локально.
1. Рекомендуется использовать флаг Docker `--init` или [dumb-init](https://github.com/Yelp/dumb-init), чтобы избежать специального обращения с процессами с PID=1. Это распространенная причина появления зомби-процессов.

### Azure Pipelines

Для агентов Windows или macOS дополнительная конфигурация не требуется, просто установите Playwright и запустите ваши тесты.

Для агентов Linux вы можете использовать [наш контейнер Docker](./docker.mdx) с поддержкой Azure Pipelines [запуска контейнеризованных заданий](https://docs.microsoft.com/en-us/azure/devops/pipelines/process/container-phases?view=azure-devops). В качестве альтернативы вы можете использовать [инструменты командной строки](./browsers#install-system-dependencies) для установки всех необходимых зависимостей.

Для запуска тестов Playwright используйте эту задачу конвейера:

```yml
trigger:
- main

pool:
  vmImage: ubuntu-latest

steps:
- task: NodeTool@0
  inputs:
    versionSpec: '18'
  displayName: 'Установить Node.js'
- script: npm ci
  displayName: 'npm ci'
- script: npx playwright install --with-deps
  displayName: 'Установить браузеры Playwright'
- script: npx playwright test
  displayName: 'Запустить тесты Playwright'
  env:
    CI: 'true'
```

#### Загрузка папки playwright-report с Azure Pipelines

Это приведет к сбою выполнения конвейера, если любой из тестов Playwright провалится. Если вы также хотите интегрировать результаты тестов с Azure DevOps, используйте задачу `PublishTestResults` следующим образом:

```yml
trigger:
- main

pool:
  vmImage: ubuntu-latest

steps:
- task: NodeTool@0
  inputs:
    versionSpec: '18'
  displayName: 'Установить Node.js'

- script: npm ci
  displayName: 'npm ci'
- script: npx playwright install --with-deps
  displayName: 'Установить браузеры Playwright'
- script: npx playwright test
  displayName: 'Запустить тесты Playwright'
  env:
    CI: 'true'
- task: PublishTestResults@2
  displayName: 'Опубликовать результаты тестов'
  inputs:
    searchFolder: 'test-results'
    testResultsFormat: 'JUnit'
    testResultsFiles: 'e2e-junit-results.xml'
    mergeTestResults: true
    failTaskOnFailedTests: true
    testRunTitle: 'Мои End-To-End тесты'
  condition: succeededOrFailed()
- task: PublishPipelineArtifact@1
  inputs:
    targetPath: playwright-report
    artifact: playwright-report
    publishLocation: 'pipeline'
  condition: succeededOrFailed()

```

Примечание: Репортер JUnit необходимо настроить соответствующим образом через

```js
import { defineConfig } from '@playwright/test';

export default defineConfig({
  reporter: [['junit', { outputFile: 'test-results/e2e-junit-results.xml' }]],
});
```

в `playwright.config.ts`.

### CircleCI

Запуск Playwright на CircleCI очень похож на запуск на GitHub Actions. Чтобы указать предварительно собранный [образ Docker](./docker.mdx) Playwright, просто измените определение агента с `docker:` в вашей конфигурации следующим образом:

```yml
executors:
  pw-noble-development:
    docker:
      - image: mcr.microsoft.com/playwright:v1.51.0-noble
```

Примечание: При использовании определения агента docker вы указываете класс ресурсов, в котором выполняется playwright, на уровне 'medium' [здесь](https://circleci.com/docs/configuration-reference?#docker-execution-environment). Поведение по умолчанию для Playwright - установить количество рабочих процессов в соответствии с обнаруженным количеством ядер (2 в случае среднего уровня). Переопределение количества рабочих процессов на большее значение приведет к ненужным тайм-аутам и сбоям.

#### Шардинг в CircleCI

Шардинг в CircleCI индексируется с 0, что означает, что вам нужно будет переопределить переменные окружения по умолчанию для параллелизма. Следующий пример демонстрирует, как запустить Playwright с параллелизмом CircleCI 4, добавив 1 к `CIRCLE_NODE_INDEX`, чтобы передать в аргумент `--shard` cli.

```yml
  playwright-job-name:
    executor: pw-noble-development
    parallelism: 4
    steps:
      - run: SHARD="$((${CIRCLE_NODE_INDEX}+1))"; npx playwright test --shard=${SHARD}/${CIRCLE_NODE_TOTAL}
```

#### Шардинг

GitLab CI поддерживает [шардинг тестов между несколькими заданиями](https://docs.gitlab.com/ee/ci/jobs/job_control.html#parallelize-large-jobs) с использованием ключевого слова [parallel](https://docs.gitlab.com/ee/ci/yaml/index.html#parallel). Тестовое задание будет разделено на несколько меньших заданий, которые выполняются параллельно. Параллельные задания именуются последовательно от `job_name 1/N` до `job_name N/N`.

```yml
stages:
  - test

tests:
  stage: test
  image: mcr.microsoft.com/playwright:v1.51.0-noble
  parallel: 7
  script:
    - npm ci
    - npx playwright test --shard=$CI_NODE_INDEX/$CI_NODE_TOTAL
```

GitLab CI также поддерживает шардирование тестов между несколькими заданиями с использованием опции [parallel:matrix](https://docs.gitlab.com/ee/ci/yaml/index.html#parallelmatrix). Тестовое задание будет выполняться несколько раз параллельно в одном конвейере, но с разными значениями переменных для каждого экземпляра задания. В приведенном ниже примере у нас есть 2 значения `PROJECT` и 10 значений `SHARD`, что приводит к общему количеству 20 выполняемых заданий.

```yml
stages:
  - test

tests:
  stage: test
  image: mcr.microsoft.com/playwright:v1.51.0-noble
  parallel:
    matrix:
      - PROJECT: ['chromium', 'webkit']
        SHARD: ['1/10', '2/10', '3/10', '4/10', '5/10', '6/10', '7/10', '8/10', '9/10', '10/10']
  script:
    - npm ci
    - npx playwright test --project=$PROJECT --shard=$SHARD
```

## Кэширование браузеров

Кэширование бинарных файлов браузеров не рекомендуется, так как время, необходимое для восстановления кэша, сопоставимо с временем, необходимым для загрузки бинарных файлов. Особенно в Linux, [зависимости операционной системы](./browsers.mdx#install-system-dependencies) необходимо установить, которые не могут быть закэшированы.

Если вы все же хотите кэшировать бинарные файлы браузеров между запусками CI, кэшируйте [эти директории](./browsers.mdx#managing-browser-binaries) в вашей конфигурации CI, основываясь на хэше версии Playwright.

## Отладка запуска браузера

Playwright поддерживает переменную окружения `DEBUG` для вывода отладочных логов во время выполнения. Установка ее в `pw:browser` полезна при отладке ошибок `Error: Failed to launch browser`.

```bash
DEBUG=pw:browser npx playwright test
```

## Запуск в обычном режиме

По умолчанию Playwright запускает браузеры в безголовом режиме. Смотрите в нашем [руководстве по запуску тестов](./running-tests.mdx#run-tests-in-headed-mode), как запустить тесты в обычном режиме.

На Linux-агентах выполнение в обычном режиме требует установки [Xvfb](https://en.wikipedia.org/wiki/Xvfb). Наш [образ Docker](./docker.mdx) и GitHub Action имеют предустановленный Xvfb. Чтобы запустить браузеры в обычном режиме с Xvfb, добавьте `xvfb-run` перед фактической командой.

```bash
xvfb-run npx playwright test
```
