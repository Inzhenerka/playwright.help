## Введение

Playwright можно использовать для тестирования вашего приложения на наличие различных проблем с доступностью.

Некоторые примеры проблем, которые могут быть обнаружены:
- Текст, который будет трудно читать для пользователей с нарушениями зрения из-за плохого цветового контраста с фоном
- Элементы управления пользовательским интерфейсом и формы без меток, которые могли бы быть идентифицированы экранным читалкой
- Интерактивные элементы с дублирующимися идентификаторами, что может запутать вспомогательные технологии

Следующие примеры используют пакет [`@axe-core/playwright`](https://npmjs.org/@axe-core/playwright), который добавляет поддержку для запуска [движка тестирования доступности axe](https://www.deque.com/axe/) в рамках ваших тестов Playwright.

:::note[Отказ от ответственности]

Автоматизированные тесты доступности могут обнаружить некоторые распространенные проблемы с доступностью, такие как отсутствующие или недействительные свойства. Однако многие проблемы с доступностью можно выявить только с помощью ручного тестирования. Мы рекомендуем использовать комбинацию автоматизированного тестирования, ручных оценок доступности и инклюзивного пользовательского тестирования.

Для ручных оценок мы рекомендуем [Accessibility Insights for Web](https://accessibilityinsights.io/docs/web/overview/?referrer=playwright-accessibility-testing-js), бесплатный и открытый инструмент для разработчиков, который поможет вам оценить веб-сайт на соответствие [WCAG 2.1 AA](https://www.w3.org/WAI/WCAG21/quickref/?currentsidebar=%23col_customize&levels=aaa).
:::

## Примеры тестов доступности

Тесты доступности работают так же, как и любые другие тесты Playwright. Вы можете либо создать отдельные тестовые случаи для них, либо интегрировать сканирование доступности и утверждения в ваши существующие тестовые случаи.

Следующие примеры демонстрируют несколько базовых сценариев тестирования доступности.

### Сканирование всей страницы

Этот пример демонстрирует, как протестировать всю страницу на наличие автоматически обнаруживаемых нарушений доступности. Тест:
1. Импортирует пакет `@axe-core/playwright`
1. Использует обычный синтаксис Playwright Test для определения тестового случая
1. Использует обычный синтаксис Playwright для навигации на тестируемую страницу
1. Ожидает выполнения `AxeBuilder.analyze()`, чтобы запустить сканирование доступности на странице
1. Использует обычные утверждения Playwright Test [assertions](./test-assertions), чтобы проверить, что в возвращенных результатах сканирования нет нарушений

### Настройка axe для сканирования конкретной части страницы

`@axe-core/playwright` поддерживает множество параметров конфигурации для axe. Вы можете указать эти параметры, используя шаблон Builder с классом `AxeBuilder`.

Например, вы можете использовать [`AxeBuilder.include()`](https://github.com/dequelabs/axe-core-npm/blob/develop/packages/playwright/README.md#axebuilderincludeselector-string--string), чтобы ограничить сканирование доступности только одной конкретной частью страницы.

`AxeBuilder.analyze()` будет сканировать страницу *в ее текущем состоянии* при вызове. Чтобы сканировать части страницы, которые становятся видимыми в зависимости от взаимодействия с пользовательским интерфейсом, используйте [Locators](./locators.mdx) для взаимодействия со страницей перед вызовом `analyze()`:

```js
test('navigation menu should not have automatically detectable accessibility violations', async ({
  page,
}) => {
  await page.goto('https://your-site.com/');

  await page.getByRole('button', { name: 'Navigation Menu' }).click();

  // Важно дождаться, пока страница будет в желаемом
  // состоянии *перед* запуском analyze(). В противном случае axe может не
  // найти все элементы, которые ваш тест ожидает, что он просканирует.
  await page.locator('#navigation-menu-flyout').waitFor();

  const accessibilityScanResults = await new AxeBuilder({ page })
      .include('#navigation-menu-flyout')
      .analyze();

  expect(accessibilityScanResults.violations).toEqual([]);
});
```

### Сканирование на наличие нарушений WCAG

По умолчанию axe проверяет на соответствие широкому спектру правил доступности. Некоторые из этих правил соответствуют конкретным критериям успеха из [Руководства по доступности веб-контента (WCAG)](https://www.w3.org/TR/WCAG21/), а другие являются правилами "лучшей практики", которые не являются строго обязательными по каким-либо критериям WCAG.

Вы можете ограничить сканирование доступности только теми правилами, которые "отмечены" как соответствующие конкретным критериям успеха WCAG, используя [`AxeBuilder.withTags()`](https://github.com/dequelabs/axe-core-npm/blob/develop/packages/playwright/README.md#axebuilderwithtagstags-stringarray). Например, [Автоматические проверки Accessibility Insights for Web](https://accessibilityinsights.io/docs/web/getstarted/fastpass/?referrer=playwright-accessibility-testing-js) включают только правила axe, которые проверяют на наличие нарушений критериев успеха WCAG A и AA; чтобы соответствовать этому поведению, вы бы использовали теги `wcag2a`, `wcag2aa`, `wcag21a` и `wcag21aa`.

Обратите внимание, что автоматизированное тестирование не может обнаружить все типы нарушений WCAG.

```js
test('should not have any automatically detectable WCAG A or AA violations', async ({ page }) => {
  await page.goto('https://your-site.com/');

  const accessibilityScanResults = await new AxeBuilder({ page })
      .withTags(['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa'])
      .analyze();

  expect(accessibilityScanResults.violations).toEqual([]);
});
```

Вы можете найти полный список тегов правил, поддерживаемых axe-core, в [разделе "Теги Axe-core" документации API axe](https://www.deque.com/axe/core-documentation/api-documentation/#axe-core-tags).

## Обработка известных проблем

Распространенный вопрос при добавлении тестов доступности в приложение: "как подавить известные нарушения?" Следующие примеры демонстрируют несколько техник, которые вы можете использовать.

### Исключение отдельных элементов из сканирования

Если ваше приложение содержит несколько конкретных элементов с известными проблемами, вы можете использовать [`AxeBuilder.exclude()`](https://github.com/dequelabs/axe-core-npm/blob/develop/packages/playwright/README.md#axebuilderexcludeselector-string--string), чтобы исключить их из сканирования до тех пор, пока вы не сможете исправить проблемы.

Это обычно самый простой вариант, но у него есть некоторые важные недостатки:
* `exclude()` исключит указанные элементы *и всех их потомков*. Избегайте использования его с компонентами, которые содержат много дочерних элементов.
* `exclude()` предотвратит выполнение *всех* правил для указанных элементов, а не только тех правил, которые соответствуют известным проблемам.

Вот пример исключения одного элемента из сканирования в одном конкретном тесте:

```js
test('should not have any accessibility violations outside of elements with known issues', async ({
  page,
}) => {
  await page.goto('https://your-site.com/page-with-known-issues');

  const accessibilityScanResults = await new AxeBuilder({ page })
      .exclude('#element-with-known-issue')
      .analyze();

  expect(accessibilityScanResults.violations).toEqual([]);
});
```

Если элемент, о котором идет речь, используется многократно на многих страницах, рассмотрите возможность [использования тестового фикстуры](#using-a-test-fixture-for-common-axe-configuration) для повторного использования одной и той же конфигурации `AxeBuilder` в нескольких тестах.

### Отключение отдельных правил сканирования

Если в вашем приложении есть много различных предшествующих нарушений конкретного правила, вы можете использовать [`AxeBuilder.disableRules()`](https://github.com/dequelabs/axe-core-npm/blob/develop/packages/playwright/README.md#axebuilderdisablerulesrules-stringarray), чтобы временно отключить отдельные правила, пока вы не сможете исправить проблемы.

Вы можете найти идентификаторы правил, которые нужно передать в `disableRules()`, в свойстве `id` нарушений, которые вы хотите подавить. Полный список правил axe можно найти в [документации `axe-core`](https://github.com/dequelabs/axe-core/blob/master/doc/rule-descriptions.md).

```js
test('should not have any accessibility violations outside of rules with known issues', async ({
  page,
}) => {
  await page.goto('https://your-site.com/page-with-known-issues');

  const accessibilityScanResults = await new AxeBuilder({ page })
      .disableRules(['duplicate-id'])
      .analyze();

  expect(accessibilityScanResults.violations).toEqual([]);
});
```

### Использование снимков для разрешения конкретных известных проблем

Если вы хотите разрешить более детализированный набор известных проблем, вы можете использовать [Снимки](./test-snapshots.mdx), чтобы проверить, что набор предшествующих нарушений не изменился. Этот подход избегает недостатков использования `AxeBuilder.exclude()`, но требует немного больше сложности и хрупкости.

Не используйте снимок всего массива `accessibilityScanResults.violations`. Он содержит детали реализации элементов, о которых идет речь, такие как фрагмент их отрисованного HTML; если вы включите их в свои снимки, это сделает ваши тесты подверженными сбоям каждый раз, когда один из компонентов изменится по не связанным причинам:

```js
// Не делайте этого! Это хрупко.
expect(accessibilityScanResults.violations).toMatchSnapshot();
```

Вместо этого создайте *отпечаток* нарушения(ий), который содержит только достаточно информации для уникальной идентификации проблемы, и используйте снимок отпечатка:

```js
// Это менее хрупко, чем создание снимка всего массива нарушений.
expect(violationFingerprints(accessibilityScanResults)).toMatchSnapshot();

// my-test-utils.js
function violationFingerprints(accessibilityScanResults) {
  const violationFingerprints = accessibilityScanResults.violations.map(violation => ({
    rule: violation.id,
    // Это CSS-селекторы, которые уникально идентифицируют каждый элемент с
    // нарушением соответствующего правила.
    targets: violation.nodes.map(node => node.target),
  }));

  return JSON.stringify(violationFingerprints, null, 2);
}
```

## Экспорт результатов сканирования как вложение теста

Большинство тестов доступности в первую очередь касаются свойства `violations` результатов сканирования axe. Однако результаты сканирования содержат больше, чем просто `violations`. Например, результаты также содержат информацию о правилах, которые прошли проверку, и о элементах, для которых axe обнаружил неопределенные результаты для некоторых правил. Эта информация может быть полезна для отладки тестов, которые не обнаруживают все нарушения, которые вы ожидаете.

Чтобы включить *все* результаты сканирования в качестве части ваших тестовых результатов для целей отладки, вы можете добавить результаты сканирования как вложение теста с помощью [`testInfo.attach()`](./api/class-testinfo#test-info-attach). [Репортеры](./test-reporters) затем могут встраивать или связывать полные результаты как часть вашего тестового вывода.

Следующий пример демонстрирует прикрепление результатов сканирования к тесту:

```js
test('example with attachment', async ({ page }, testInfo) => {
  await page.goto('https://your-site.com/');

  const accessibilityScanResults = await new AxeBuilder({ page }).analyze();

  await testInfo.attach('accessibility-scan-results', {
    body: JSON.stringify(accessibilityScanResults, null, 2),
    contentType: 'application/json'
  });

  expect(accessibilityScanResults.violations).toEqual([]);
});
```

## Использование тестовой фикстуры для общей конфигурации axe

[Testовые фикстуры](./test-fixtures) — это хороший способ поделиться общей конфигурацией `AxeBuilder` между многими тестами. Некоторые сценарии, когда это может быть полезно, включают:
* Использование общего набора правил среди всех ваших тестов
* Подавление известного нарушения в общем элементе, который появляется на многих разных страницах
* Последовательное прикрепление отдельных отчетов о доступности для многих сканирований

Следующий пример демонстрирует создание и использование тестовой фикстуры, которая охватывает каждый из этих сценариев.

### Создание фикстуры

Эта примерная фикстура создает объект `AxeBuilder`, который предварительно настроен с общей конфигурацией `withTags()` и `exclude()`.

<Tabs
  groupId="js-flavor"
  defaultValue="ts"

```js title="axe-test.ts"
import { test as base } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

type AxeFixture = {
  makeAxeBuilder: () => AxeBuilder;
};

// Расширяем базовый тест, предоставляя "makeAxeBuilder"
//
// Этот новый "тест" может использоваться в нескольких тестовых файлах, и каждый из них получит
// последовательно настроенный экземпляр AxeBuilder.
export const test = base.extend<AxeFixture>({
  makeAxeBuilder: async ({ page }, use, testInfo) => {
    const makeAxeBuilder = () => new AxeBuilder({ page })
        .withTags(['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa'])
        .exclude('#commonly-reused-element-with-known-issue');

    await use(makeAxeBuilder);
  }
});
export { expect } from '@playwright/test';
```

</TabItem>

const base = require('@playwright/test');
const AxeBuilder = require('@axe-core/playwright').default;

### Использование фикстуры

Чтобы использовать фикстуру, замените `new AxeBuilder({ page })` в предыдущих примерах на вновь определенную фикстуру `makeAxeBuilder`:

```js
const { test, expect } = require('./axe-test');

test('example using custom fixture', async ({ page, makeAxeBuilder }) => {
  await page.goto('https://your-site.com/');

  const accessibilityScanResults = await makeAxeBuilder()
      // Автоматически использует общую конфигурацию AxeBuilder,
      // но также поддерживает дополнительную конфигурацию, специфичную для теста
      .include('#specific-element-under-test')
      .analyze();

  expect(accessibilityScanResults.violations).toEqual([]);
});
```
