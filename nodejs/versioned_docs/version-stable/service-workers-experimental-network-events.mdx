## Введение

:::warning

Если вы ищете общие методы для мокирования, маршрутизации и перехвата сетевых запросов, сначала ознакомьтесь с [Руководством по сети](./network.mdx). Playwright предоставляет встроенные API для этого случая, которые не требуют информации ниже. Однако, если вас интересуют запросы, сделанные самими Service Workers, пожалуйста, читайте дальше.
:::

[Service Workers](https://developer.mozilla.org/en-US/docs/Web/API/Service_Worker_API) предоставляют нативный метод браузера для обработки запросов, сделанных страницей с использованием нативного [Fetch API (`fetch`)](https://developer.mozilla.org/en-US/docs/Web/API/Fetch_API), а также других сетевых ресурсов (таких как скрипты, css и изображения).

Они могут действовать как **сетевой прокси** между страницей и внешней сетью для выполнения логики кэширования или могут предоставить пользователям оффлайн-опыт, если Service Worker добавляет слушатель [FetchEvent](https://developer.mozilla.org/en-US/docs/Web/API/FetchEvent#examples).

Многие сайты, использующие Service Workers, просто используют их как прозрачную технику оптимизации. Хотя пользователи могут заметить более быструю работу, реализация приложения не осознает их существования. Запуск приложения с включенными или отключенными Service Workers выглядит функционально эквивалентным.

## Как включить

Инспекция и маршрутизация запросов, сделанных Service Workers в Playwright, являются **экспериментальными** и отключены по умолчанию.

Установите переменную окружения `PW_EXPERIMENTAL_SERVICE_WORKER_NETWORK_EVENTS` в `1` (или любое другое значение), чтобы включить эту функцию. В настоящее время поддерживаются только Chrome/Chromium.

Если вы используете (или заинтересованы в использовании этой функции), пожалуйста, прокомментируйте [эту проблему](https://github.com/microsoft/playwright/issues/15684), сообщив нам о вашем случае использования.

## Fetch Service Worker

### Доступ к Service Workers и ожидание активации

Вы можете использовать [browserContext.serviceWorkers()](/api/class-browsercontext.mdx#browser-context-service-workers), чтобы перечислить Service [Worker]ы, или специально отслеживать Service [Worker], если вы ожидаете, что страница вызовет его [регистрацию](https://developer.mozilla.org/en-US/docs/Web/API/ServiceWorkerContainer/register):

```js
const serviceWorkerPromise = context.waitForEvent('serviceworker');
await page.goto('/example-with-a-service-worker.html');
const serviceworker = await serviceWorkerPromise;
```

[browserContext.on('serviceworker')](/api/class-browsercontext.mdx#browser-context-event-service-worker) срабатывает ***до*** того, как основной скрипт Service Worker будет оценен, поэтому ***прежде чем*** вызывать service[worker.evaluate()](/api/class-worker.mdx#worker-evaluate), вам следует дождаться его активации.

Существуют более идиоматические методы ожидания активации Service Worker, но следующий метод является независимым от реализации:

```js
await page.evaluate(async () => {
  const registration = await window.navigator.serviceWorker.getRegistration();
  if (registration.active?.state === 'activated')
    return;
  await new Promise(res =>
    window.navigator.serviceWorker.addEventListener('controllerchange', res),
  );
});
```

### Сетевые события и маршрутизация

Любой сетевой запрос, сделанный **Service Worker**, будет иметь:
* [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) и соответствующие события ([browserContext.on('requestfinished')](/api/class-browsercontext.mdx#browser-context-event-request-finished) и [browserContext.on('response')](/api/class-browsercontext.mdx#browser-context-event-response), или [browserContext.on('requestfailed')](/api/class-browsercontext.mdx#browser-context-event-request-failed))
* [browserContext.route()](/api/class-browsercontext.mdx#browser-context-route) увидит запрос
* [request.serviceWorker()](/api/class-request.mdx#request-service-worker) будет установлен на экземпляр Service [Worker], а [request.frame()](/api/class-request.mdx#request-frame) **выдаст ошибку**
* [response.fromServiceWorker()](/api/class-response.mdx#response-from-service-worker) вернет `false`

Кроме того, любой сетевой запрос, сделанный **Страницей** (включая ее под-[Frame]ы), будет иметь:
* [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) и соответствующие события ([browserContext.on('requestfinished')](/api/class-browsercontext.mdx#browser-context-event-request-finished) и [browserContext.on('response')](/api/class-browsercontext.mdx#browser-context-event-response), или [browserContext.on('requestfailed')](/api/class-browsercontext.mdx#browser-context-event-request-failed))
* [page.on('request')](/api/class-page.mdx#page-event-request) и соответствующие события ([page.on('requestfinished')](/api/class-page.mdx#page-event-request-finished) и [page.on('response')](/api/class-page.mdx#page-event-response), или [page.on('requestfailed')](/api/class-page.mdx#page-event-request-failed))
* [page.route()](/api/class-page.mdx#page-route) и [page.route()](/api/class-page.mdx#page-route) **не увидят** запрос (если был зарегистрирован обработчик fetch Service Worker)
* [request.serviceWorker()](/api/class-request.mdx#request-service-worker) будет установлен на `null`, а [request.frame()](/api/class-request.mdx#request-frame) вернет [Frame]
* [response.fromServiceWorker()](/api/class-response.mdx#response-from-service-worker) вернет `true` (если был зарегистрирован обработчик fetch Service Worker)

Многие реализации Service Worker просто выполняют запрос со стороны страницы (возможно, с некоторой пользовательской логикой кэширования/оффлайн, опущенной для простоты):

```js title="transparent-service-worker.js"
self.addEventListener('fetch', event => {
  // фактически выполняет запрос
  const responsePromise = fetch(event.request);
  // отправляет его обратно на страницу
  event.respondWith(responsePromise);
});

self.addEventListener('activate', event => {
  event.waitUntil(clients.claim());
});
```

Если страница регистрирует вышеуказанный Service Worker:

```html
<!-- filename: index.html -->
<script>
  window.registrationPromise = navigator.serviceWorker.register('/transparent-service-worker.js');
</script>
```

При первом посещении страницы через [page.goto()](/api/class-page.mdx#page-goto) будут сгенерированы следующие события Request/Response (вместе с соответствующими сетевыми событиями жизненного цикла):

| Событие                           | Владелец         | URL                            | Маршрутизировано | [response.fromServiceWorker()](/api/class-response.mdx#response-from-service-worker) |
| -                                 | -                | -                              | -      | -                                      |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | [Frame]          | index.html                     | Да     |                                        |
| [page.on('request')](/api/class-page.mdx#page-event-request)           | [Frame]          | index.html                     | Да     |                                        |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | Service [Worker] | transparent-service-worker.js  | Да     |                                        |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | Service [Worker] | data.json                      | Да     |                                        |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | [Frame]          | data.json                      |        | Да                                    |
| [page.on('request')](/api/class-page.mdx#page-event-request)           | [Frame]          | data.json                      |        | Да                                    |

Поскольку пример Service Worker просто действует как базовый прозрачный "прокси":
* Существует 2 события [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) для `data.json`; одно принадлежит [Frame], другое - Service [Worker].
* Только запрос, принадлежащий Service [Worker], для ресурса был маршрутизируемым через [browserContext.route()](/api/class-browsercontext.mdx#browser-context-route); события, принадлежащие [Frame], для `data.json` не маршрутизируемы, так как они даже не имели возможности выйти в внешнюю сеть, поскольку у Service Worker зарегистрирован обработчик fetch.

:::caution

Важно отметить: вызов [request.frame()](/api/class-request.mdx#request-frame) или [response.frame()](/api/class-response.mdx#response-frame) **выдаст** исключение, если вызван на [Request]/[Response], у которого [request.serviceWorker()](/api/class-request.mdx#request-service-worker) не равен null.
:::

#### Расширенный пример

Когда Service Worker обрабатывает запрос страницы, он может сделать от 0 до n запросов к внешней сети. Service Worker может ответить непосредственно из кэша, сгенерировать ответ в памяти, переписать запрос, сделать два запроса и затем объединить их в один и т.д.

Рассмотрите приведенные ниже фрагменты кода, чтобы понять, как Playwright видит Request/Responses и как это влияет на маршрутизацию в некоторых из этих случаев.

```js title="complex-service-worker.js"
self.addEventListener('install', function(event) {
  event.waitUntil(
      caches.open('v1').then(function(cache) {
      // 1. Предварительно загружает и кэширует /addressbook.json
        return cache.add('/addressbook.json');
      })
  );
});

// Выбор обработки FetchEvent'ов со стороны страницы
self.addEventListener('fetch', event => {
  event.respondWith(
      (async () => {
        // 1. Сначала попробуйте обслужить непосредственно из кэшей
        const response = await caches.match(event.request);
        if (response)
          return response;

        // 2. Перепишите запрос с /foo на /bar
        if (event.request.url.endsWith('foo'))
          return fetch('./bar');

        // 3. Запретите получение tracker.js и верните ответ-заполнитель
        if (event.request.url.endsWith('tracker.js')) {
          return new Response('console.log("no trackers!")', {
            status: 200,
            headers: { 'Content-Type': 'text/javascript' },
          });
        }

        // 4. В противном случае, выполните fetch и ответьте
        return fetch(event.request);
      })()
  );
});

self.addEventListener('activate', event => {
  event.waitUntil(clients.claim());
});
```

И страница, которая просто регистрирует Service Worker:

```html
<!-- filename: index.html -->
<script>
  window.registrationPromise = navigator.serviceWorker.register('/complex-service-worker.js');
</script>
```

При первом посещении страницы через [page.goto()](/api/class-page.mdx#page-goto) будут сгенерированы следующие события Request/Response:

| Событие                           | Владелец         | URL                            | Маршрутизировано | [response.fromServiceWorker()](/api/class-response.mdx#response-from-service-worker) |
| -                                 | -                | -                              | -      | -                                      |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | [Frame]          | index.html                     | Да     |                                        |
| [page.on('request')](/api/class-page.mdx#page-event-request)           | [Frame]          | index.html                     | Да     |                                        |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | Service [Worker] | complex-service-worker.js      | Да     |                                        |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | Service [Worker] | addressbook.json               | Да     |                                        |

Важно отметить, что [`cache.add`](https://developer.mozilla.org/en-US/docs/Web/API/Cache/add) вызвало запрос Service Worker (принадлежащий Service [Worker]), даже до того, как `addressbook.json` был запрошен на странице.

После активации Service Worker и обработки FetchEvents, если страница делает следующие запросы:

```js
await page.evaluate(() => fetch('/addressbook.json'));
await page.evaluate(() => fetch('/foo'));
await page.evaluate(() => fetch('/tracker.js'));
await page.evaluate(() => fetch('/fallthrough.txt'));
```

Будут сгенерированы следующие события Request/Response:

| Событие                           | Владелец         | URL                            | Маршрутизировано | [response.fromServiceWorker()](/api/class-response.mdx#response-from-service-worker) |
| -                                 | -                | -                              | -      | -                                      |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | [Frame]          | addressbook.json               |        | Да                                    |
| [page.on('request')](/api/class-page.mdx#page-event-request)           | [Frame]          | addressbook.json               |        | Да                                    |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | Service [Worker] | bar                            | Да     |                                        |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | [Frame]          | foo                            |        | Да                                    |
| [page.on('request')](/api/class-page.mdx#page-event-request)           | [Frame]          | foo                            |        | Да                                    |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | [Frame]          | tracker.js                     |        | Да                                    |
| [page.on('request')](/api/class-page.mdx#page-event-request)           | [Frame]          | tracker.js                     |        | Да                                    |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | Service [Worker] | fallthrough.txt                | Да     |                                        |
| [browserContext.on('request')](/api/class-browsercontext.mdx#browser-context-event-request) | [Frame]          | fallthrough.txt                |        | Да                                    |
| [page.on('request')](/api/class-page.mdx#page-event-request)           | [Frame]          | fallthrough.txt                |        | Да                                    |

Важно отметить:
* Страница запрашивала `/foo`, но Service Worker запрашивал `/bar`, поэтому существуют только события, принадлежащие [Frame], для `/foo`, но не для `/bar`.
* Точно так же Service Worker никогда не выходил в сеть для `tracker.js`, поэтому только события, принадлежащие [Frame], были сгенерированы для этого запроса.

## Маршрутизация только запросов Service Worker

```js
await context.route('**', async route => {
  if (route.request().serviceWorker()) {
    // Примечание: вызов route.request().frame() здесь вызовет ИСКЛЮЧЕНИЕ
    return route.fulfill({
      contentType: 'text/plain',
      status: 200,
      body: 'from sw',
    });
  } else {
    return route.continue();
  }
});
```

## Известные ограничения

Запросы на обновленный основной скрипт Service Worker в настоящее время не могут быть маршрутизированы (https://github.com/microsoft/playwright/issues/14711).
