## Введение

Веб-API обычно реализуются в виде HTTP-эндпоинтов. Playwright предоставляет API для **мокирования** и **модификации** сетевого трафика, как HTTP, так и HTTPS. Любые запросы, которые выполняет страница, включая [XHR](https://developer.mozilla.org/en-US/docs/Web/API/XMLHttpRequest) и [fetch](https://developer.mozilla.org/en-US/docs/Web/API/Fetch_API) запросы, могут быть отслежены, изменены и замокированы. С помощью Playwright вы также можете мокировать, используя HAR-файлы, которые содержат несколько сетевых запросов, сделанных страницей.

## Мокирование API запросов

Следующий код перехватит все вызовы к `*/**/api/v1/fruits` и вернет пользовательский ответ вместо этого. Запросы к API не будут выполнены. Тест переходит по URL, который использует замокированный маршрут, и проверяет, что мокированные данные присутствуют на странице.

```js
test("мокирует фрукт и не вызывает api", async ({ page }) => {
  // Мокируем вызов api перед навигацией
  await page.route('*/**/api/v1/fruits', async route => {
    const json = [{ name: 'Strawberry', id: 21 }];
    await route.fulfill({ json });
  });
  // Переходим на страницу
  await page.goto('https://demo.playwright.dev/api-mocking');

  // Проверяем, что фрукт Strawberry виден
  await expect(page.getByText('Strawberry')).toBeVisible();
});
```

Вы можете увидеть из трассировки примера теста, что API никогда не вызывался, однако он был выполнен с мокированными данными. ![трассировка мокирования api](https://github.com/microsoft/playwright/assets/13063165/3dc14cbf-c100-4efc-ac21-d7b52d698b53)

Читать далее о [расширенной сети](./network.mdx).

## Модификация ответов API

Иногда необходимо выполнить запрос к API, но ответ нужно изменить, чтобы обеспечить воспроизводимость тестирования. В этом случае, вместо мокирования запроса, можно выполнить запрос и выполнить его с измененным ответом.

В следующем примере мы перехватываем вызов к API фруктов и добавляем новый фрукт под названием 'Loquat' к данным. Затем мы переходим по URL и проверяем, что эти данные там:

```js
test('получает json из api и добавляет новый фрукт', async ({ page }) => {
  // Получаем ответ и добавляем к нему
  await page.route('*/**/api/v1/fruits', async route => {
    const response = await route.fetch();
    const json = await response.json();
    json.push({ name: 'Loquat', id: 100 });
    // Выполняем с использованием оригинального ответа, при этом изменяя тело ответа
    // с данным JSON объектом.
    await route.fulfill({ response, json });
  });

  // Переходим на страницу
  await page.goto('https://demo.playwright.dev/api-mocking');

  // Проверяем, что новый фрукт виден
  await expect(page.getByText('Loquat', { exact: true })).toBeVisible();
});
```

В трассировке нашего теста мы можем увидеть, что API был вызван, и ответ был изменен. ![трассировка теста, показывающая вызов и выполнение api](https://github.com/microsoft/playwright/assets/13063165/8b8dd82d-1b3e-428e-871b-840581fed439)

Проверяя ответ, мы можем увидеть, что наш новый фрукт был добавлен в список. ![трассировка теста, показывающая мокированный ответ](https://github.com/microsoft/playwright/assets/13063165/03e6c87c-4ecc-47e8-9ca0-30fface25e9d)

Читать далее о [расширенной сети](./network.mdx).

## Мокирование с помощью HAR файлов

HAR файл — это [HTTP Archive](http://www.softwareishard.com/blog/har-12-spec/) файл, который содержит запись всех сетевых запросов, сделанных при загрузке страницы. Он содержит информацию о заголовках запросов и ответов, куках, содержимом, временных интервалах и многом другом. Вы можете использовать HAR файлы для мокирования сетевых запросов в ваших тестах. Вам нужно:
1. Записать HAR файл.
1. Зафиксировать HAR файл вместе с тестами.
1. Направить запросы, используя сохраненные HAR файлы в тестах.

### Запись HAR файла

Чтобы записать HAR файл, мы используем метод [page.routeFromHAR()](/api/class-page.mdx#page-route-from-har) или [browserContext.routeFromHAR()](/api/class-browsercontext.mdx#browser-context-route-from-har). Этот метод принимает путь к HAR файлу и необязательный объект параметров. Объект параметров может содержать URL, чтобы только запросы с URL, соответствующим указанному шаблону glob, обслуживались из HAR файла. Если не указано, все запросы будут обслуживаться из HAR файла.

Установка параметра `update` в true создаст или обновит HAR файл с фактической сетевой информацией вместо обслуживания запросов из HAR файла. Используйте его при создании теста, чтобы заполнить HAR реальными данными.

```js
test('записывает или обновляет HAR файл', async ({ page }) => {
  // Получаем ответ из HAR файла
  await page.routeFromHAR('./hars/fruit.har', {
    url: '*/**/api/v1/fruits',
    update: true,
  });

  // Переходим на страницу
  await page.goto('https://demo.playwright.dev/api-mocking');

  // Проверяем, что фрукт виден
  await expect(page.getByText('Strawberry')).toBeVisible();
});
```

### Модификация HAR файла

После того как вы записали HAR файл, вы можете изменить его, открыв хэшированный .txt файл внутри вашей папки 'hars' и отредактировав JSON. Этот файл должен быть зафиксирован в вашей системе контроля версий. Каждый раз, когда вы запускаете этот тест с `update: true`, он обновит ваш HAR файл с запросом от API.

```json
[
  {
    "name": "Playwright",
    "id": 100
  },
  // ... другие фрукты
]
```

### Воспроизведение из HAR

Теперь, когда у вас есть записанный и измененный HAR файл с мокированными данными, его можно использовать для обслуживания соответствующих ответов в тесте. Для этого просто отключите или просто удалите параметр `update`. Это запустит тест против HAR файла вместо вызова API.

```js
test('получает json из HAR и проверяет, что новый фрукт был добавлен', async ({ page }) => {
  // Воспроизводим API запросы из HAR.
  // Либо используйте соответствующий ответ из HAR,
  // либо отмените запрос, если ничего не соответствует.
  await page.routeFromHAR('./hars/fruit.har', {
    url: '*/**/api/v1/fruits',
    update: false,
  });

  // Переходим на страницу
  await page.goto('https://demo.playwright.dev/api-mocking');

  // Проверяем, что фрукт Playwright виден
  await expect(page.getByText('Playwright', { exact: true })).toBeVisible();
});
```

В трассировке нашего теста мы можем увидеть, что маршрут был выполнен из HAR файла, и API не был вызван. ![трассировка, показывающая использование HAR файла](https://github.com/microsoft/playwright/assets/13063165/1bd7ab66-ea4f-43c2-a4e5-ca17d4837ff1)

Если мы проверим ответ, мы увидим, что наш новый фрукт был добавлен в JSON, что было сделано путем ручного обновления хэшированного `.txt` файла внутри папки `hars`. ![трассировка, показывающая ответ из HAR файла](https://github.com/microsoft/playwright/assets/13063165/db3117fc-7b02-4973-9a51-29e213261a6a)

Воспроизведение HAR строго соответствует URL и методу HTTP. Для POST запросов также строго соответствует полезной нагрузке POST. Если несколько записей соответствуют запросу, выбирается та, у которой больше всего совпадающих заголовков. Запись, приводящая к перенаправлению, будет автоматически выполнена.

Аналогично записи, если имя HAR файла заканчивается на `.zip`, он считается архивом, содержащим HAR файл вместе с сетевыми полезными нагрузками, хранящимися как отдельные записи. Вы также можете извлечь этот архив, отредактировать полезные нагрузки или журнал HAR вручную и указать на извлеченный har файл. Все полезные нагрузки будут разрешены относительно извлеченного har файла в файловой системе.

#### Запись HAR с помощью CLI

Мы рекомендуем использовать параметр `update` для записи HAR файла для вашего теста. Однако вы также можете записать HAR с помощью Playwright CLI.

Откройте браузер с помощью Playwright CLI и передайте параметр `--save-har`, чтобы создать HAR файл. При желании используйте `--save-har-glob`, чтобы сохранить только интересующие вас запросы, например, конечные точки API. Если имя har файла заканчивается на `.zip`, артефакты записываются как отдельные файлы и все сжимаются в один `zip`.

```bash
# Сохранить API запросы с example.com как архив "example.har".
npx playwright open --save-har=example.har --save-har-glob="**/api/**" https://example.com
```

## Мокирование WebSockets

Следующий код перехватит соединения WebSocket и замокирует всю коммуникацию через WebSocket, вместо подключения к серверу. Этот пример отвечает на `"request"` с `"response"`.

```js
await page.routeWebSocket('wss://example.com/ws', ws => {
  ws.onMessage(message => {
    if (message === 'request')
      ws.send('response');
  });
});
```

В качестве альтернативы, вы можете захотеть подключиться к реальному серверу, но перехватить сообщения между ними и изменить или заблокировать их. Вот пример, который изменяет некоторые сообщения, отправленные страницей на сервер, и оставляет остальные без изменений.

```js
await page.routeWebSocket('wss://example.com/ws', ws => {
  const server = ws.connectToServer();
  ws.onMessage(message => {
    if (message === 'request')
      server.send('request2');
    else
      server.send(message);
  });
});
```

Для получения дополнительной информации смотрите [WebSocketRoute].
